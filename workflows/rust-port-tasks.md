# Rust Port Task Specifications

<!-- markdownlint-disable-file -->

These are the task assignments for the `rust-port` workflow. Run them **in order** — each phase builds on the previous.

## Execution

Each task is dispatched through the `rust-port.workflow.json` workflow as a separate assignment. The workflow handles branching, QA gating, and merge to main per-phase.

```bash
# From the a_c_a/ directory, dispatch each task in sequence:
# (Replace the launcher command with your actual ACA invocation)

# Phase 1: Must complete first — everything depends on the C shim
aca run --workflow workflows/rust-port.workflow.json --task-id rust-ffi-shim

# Phase 2: Depends on Phase 1 (links against the C shim)
aca run --workflow workflows/rust-port.workflow.json --task-id rust-sys-crate

# Phase 3: Depends on Phase 2 (uses the safe Rust wrapper)
aca run --workflow workflows/rust-port.workflow.json --task-id rust-example-workload

# Phase 4: Independent — can run after Phase 1 or in parallel with 2/3
aca run --workflow workflows/rust-port.workflow.json --task-id rust-flatbuffers-codegen

# Phase 5: Independent — can run any time
aca run --workflow workflows/rust-port.workflow.json --task-id rust-devcontainer
```

## Phase 1: C FFI Shim (`rust-ffi-shim`)

**Priority**: Critical — all Rust phases depend on this  
**Estimated scope**: ~15 files, ~1500 LOC  
**Dependencies**: None (only needs existing C++ SDK)

### Objective

Create a C-linkage shared library (`liblogixedge_ffi.so`) that wraps the LogixEdge C++ SDK public API with stable C ABI function signatures.

### Work Items

1. Create `core/ffi/CMakeLists.txt`:
   - New shared library target `logixedge_ffi` linking `LogixEdgeCore`
   - Install the `.so` and public header to `build/<preset>/lib` and `build/<preset>/include`
   - Include in the build via `add_subdirectory(ffi)` from `core/CMakeLists.txt`

2. Create `core/ffi/include/logixedge_ffi.h`:
   - `#ifdef __cplusplus extern "C" {` guard
   - Opaque handle types: `EdgeContext`, `EdgeWorkload`, `EdgeIpcClient`, `EdgePublication`, `EdgeSubscription`, `EdgeCallProxy`, `EdgeCallHandler`, `EdgeKeyValueStorage`, `EdgeLogger`, `EdgeTimer`, `EdgeEvent`
   - Lifecycle functions: `edge_<type>_create(...)`, `edge_<type>_destroy(...)`
   - Core operations mirroring the C++ API (publish, subscribe, call, handle, get, put, log, etc.)
   - Error handling: `EdgeResult` enum (OK, ERR_NULL, ERR_INIT, ERR_IPC, ERR_TIMEOUT, ERR_UNKNOWN) returned from every function
   - `edge_last_error()` returning a `const char*` thread-local error string

3. Create `core/ffi/src/context_ffi.cpp`, `workload_ffi.cpp`, `ipc_client_ffi.cpp`, `publication_ffi.cpp`, `subscription_ffi.cpp`, `call_proxy_ffi.cpp`, `call_handler_ffi.cpp`, `kv_storage_ffi.cpp`, `logger_ffi.cpp`, `timer_ffi.cpp`, `event_ffi.cpp`:
   - Each wraps the corresponding C++ class methods in `extern "C"` functions
   - Use try/catch to convert C++ exceptions to `EdgeResult` error codes
   - Store exception messages in thread-local `edge_last_error()` buffer
   - Use the **non-templated raw-data** callback overloads on IpcClient (the ones taking `const uint8_t*, size_t`) to avoid template instantiation across the ABI boundary

4. Create `core/ffi/test/ffi_test.cpp`:
   - Google Test suite testing every function through the C API
   - Round-trip tests: create context → create workload → create ipc client → publish → subscribe → verify receipt
   - Error path tests: null handles, double-free, operations on destroyed handles

### Success Criteria
- `cmake --build build/debug/host --target logixedge_ffi` succeeds
- `liblogixedge_ffi.so` is in `build/debug/host/lib/`
- `logixedge_ffi.h` is in `build/debug/host/include/`
- All Google Test tests pass
- No compiler warnings with `-Wall -Wextra`

## Phase 2: Rust Sys-Crate and Safe Wrapper (`rust-sys-crate`)

**Priority**: Critical — example workload depends on this  
**Estimated scope**: ~10 files, ~1200 LOC  
**Dependencies**: Phase 1 (links against `liblogixedge_ffi.so`)

### Objective

Create a Cargo workspace with a `logixedge-sys` crate (raw FFI bindings generated by `bindgen`) and a `logixedge` crate (safe Rust wrapper with RAII handles).

### Work Items

1. Create `logix-edge-rs/Cargo.toml` (workspace root):
   ```toml
   [workspace]
   resolver = "2"
   members = ["logixedge-sys", "logixedge"]
   ```

2. Create `logix-edge-rs/logixedge-sys/`:
   - `Cargo.toml` with `bindgen` as build-dependency, `links = "logixedge_ffi"`
   - `build.rs` that:
     - Reads `LOGIXEDGE_FFI_DIR` env var (or discovers from `../../build/debug/host`)
     - Runs `bindgen` on `logixedge_ffi.h`
     - Emits `cargo:rustc-link-lib=logixedge_ffi` and `cargo:rustc-link-search=...`
   - `src/lib.rs` with `include!(concat!(env!("OUT_DIR"), "/bindings.rs"));` and re-exports

3. Create `logix-edge-rs/logixedge/`:
   - `Cargo.toml` depending on `logixedge-sys`
   - `src/lib.rs` with modules: `context`, `workload`, `ipc_client`, `publication`, `subscription`, `call_proxy`, `call_handler`, `kv_storage`, `logger`, `timer`, `event`, `error`
   - `src/error.rs`: `EdgeError` enum mapping from `EdgeResult`, implements `std::error::Error`
   - `src/context.rs`: `Context` wrapping `EdgeContext*`, implements `Drop`
   - Similarly for every handle type
   - Every wrapper method converts `EdgeResult` errors to `Result<T, EdgeError>`
   - Every `unsafe` FFI call has a `// SAFETY:` comment

4. Create `logix-edge-rs/logixedge/tests/integration_test.rs`:
   - Test Context creation and teardown
   - Test Workload lifecycle
   - Test IpcClient pub/sub round-trip through safe API
   - Test error propagation (null context, etc.)

### Success Criteria
- `cd logix-edge-rs && cargo build --workspace` succeeds
- `cargo test --workspace` passes all tests
- `cargo clippy --workspace -- -D warnings` clean
- `cargo fmt --check` clean
- Zero use of `unsafe` outside `logixedge-sys` and the internal FFI call sites in `logixedge`

## Phase 3: Example Rust Workload (`rust-example-workload`)

**Priority**: High — demonstrates the full Rust SDK workflow  
**Estimated scope**: ~5 files, ~300 LOC  
**Dependencies**: Phase 2 (uses the `logixedge` crate)

### Objective

Create a minimal "hello world" Rust workload that registers with the Edge Data Router, publishes a heartbeat message, subscribes to its own topic, and logs received messages.

### Work Items

1. Add `logix-edge-rs/examples/hello-workload/` to the Cargo workspace:
   ```toml
   # In logix-edge-rs/Cargo.toml
   members = ["logixedge-sys", "logixedge", "examples/hello-workload"]
   ```

2. Create `logix-edge-rs/examples/hello-workload/`:
   - `Cargo.toml` depending on `logixedge`
   - `src/main.rs`:
     - Creates `Context` and `Workload`
     - Creates `IpcClient`, registers endpoints
     - Creates `Publication` on `/rust/hello/heartbeat`
     - Creates `Subscription` on `/rust/hello/heartbeat`
     - Publishes a simple FlatBuffer or raw bytes every second via `Timer`
     - Logs received messages with `Logger`
     - Clean shutdown on SIGTERM

3. Create `logix-edge-rs/examples/hello-workload/README.md`:
   - Build instructions
   - How to run alongside the Edge Data Router

### Success Criteria
- `cargo build -p hello-workload` succeeds
- Binary starts, logs "Hello from Rust workload", publishes, receives its own message, shuts down cleanly
- `cargo clippy -p hello-workload -- -D warnings` clean

## Phase 4: FlatBuffers Codegen Extension (`rust-flatbuffers-codegen`)

**Priority**: Medium — needed for production workloads but not for the FFI layer  
**Estimated scope**: ~3 files, ~150 LOC  
**Dependencies**: Phase 1 (conceptually, needs build system functional)

### Objective

Extend the existing `cmake/generateEdgeFlatbuffer.cmake` module to optionally emit Rust code from `.fbs` schemas alongside the existing C++ and Python outputs.

### Work Items

1. Modify `cmake/generateEdgeFlatbuffer.cmake`:
   - Add an optional `RUST` keyword argument to `flatbuffers_generate_headers()`
   - When `RUST` is specified, also invoke `flatc --rust -o <rust_output_dir> <schema>.fbs`
   - Default `rust_output_dir` to `${CMAKE_CURRENT_BINARY_DIR}/flatbuffers/rust/`
   - Create a CMake variable `${NAME}_RUST_OUTPUTS` listing generated `.rs` files

2. Create `cmake/test/test_flatbuffers_rust.cmake`:
   - CTest that generates Rust from a sample `.fbs` schema
   - Verifies the `.rs` file exists and contains expected struct names

3. Document the `RUST` option in a comment block at the top of the function

### Success Criteria
- Existing C++/Python FlatBuffers generation still works unchanged
- `flatbuffers_generate_headers(... RUST)` produces valid `.rs` files
- Generated Rust code compiles with `rustc`

## Phase 5: Dev Container Rust Toolchain (`rust-devcontainer`)

**Priority**: Medium — enables development but not blocking  
**Estimated scope**: ~2 files, ~30 LOC  
**Dependencies**: None

### Objective

Add `rustup`, `cargo`, `clippy`, and `rustfmt` to the dev container image so Rust development works out of the box.

### Work Items

1. Modify `docker/internal/Dockerfile`:
   - Add a multi-stage build that compiles the QNX Rust toolchain from the
     [qnx-ports/rust](https://github.com/qnx-ports/rust) fork (branch `qnx-1.90.0`).
     QNX is a Tier 3 Rust target — upstream `rust:*-slim` images do not include
     QNX targets. See
     [qnx-ports build instructions](https://github.com/qnx-ports/build-files/blob/main/ports/rust/README.md).
   - The build stage must:
     - Install `rustup` from upstream
     - Clone `qnx-ports/rust` at tag/branch `qnx-1.90.0`
     - Apply the QNX 8.0 config (`qnx800-config.toml` from `qnx-ports/build-files`)
     - Run `./x.py install` to produce a stage directory
   - Copy the built toolchain into the final image:
     ```dockerfile
     COPY --from=rust-qnx-builder /workspace/rust/stage/host/linux/x86_64/usr/ /usr/local/qnx-rust/
     ENV PATH=/usr/local/qnx-rust/bin:$PATH
     ```
   - Register the toolchain: `rustup toolchain link qnx-rust /usr/local/qnx-rust/`
   - Install host-target components for local dev: `rustup component add clippy rustfmt`
   - Install `bindgen` build dependencies: `libclang-dev`

2. Update `.devcontainer/devcontainer.json`:
   - Add `rust-analyzer` to VS Code extensions list

### Success Criteria
- `docker build` succeeds
- Inside container: `rustc --version` reports **1.90.0**, `cargo --version` works
- `rustup target list --installed` includes `aarch64-unknown-nto-qnx800` and `x86_64-pc-nto-qnx800`
- `clippy-driver --version`, `rustfmt --version` work
- `bindgen` can generate bindings (test with `echo '#include <stdio.h>' | bindgen -`)

## Dependency Graph

```
Phase 5 (devcontainer) ──────────────────────────────────────┐
                                                              │
Phase 1 (FFI shim) ──→ Phase 2 (sys-crate) ──→ Phase 3 (example) ──→ DONE
          │                                                   │
          └──→ Phase 4 (flatbuffers codegen) ─────────────────┘
```

**Parallelization**: Phase 5 can run at any time. Phase 4 can run after Phase 1 or in parallel with Phases 2/3. The critical path is 1 → 2 → 3.
